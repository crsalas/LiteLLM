# LiteLLM Local Proxy Base (macOS)

Reusable local LiteLLM proxy for multiple providers behind one OpenAI-compatible interface.

## What this gives you

- One endpoint for all supported providers: `http://127.0.0.1:4000/v1`
- One client contract for any project:
  - `OPENAI_BASE_URL=http://127.0.0.1:4000/v1`
  - `OPENAI_API_KEY=<LITELLM_MASTER_KEY>`
- Stable model aliases for OpenAI, Anthropic, Gemini API, and local Ollama

## Security defaults in this repo

- Local-only bind: `127.0.0.1:4000`
- Container runs as non-root host UID/GID
- Pinned LiteLLM image by immutable digest
- Read-only root filesystem + dropped Linux caps + no-new-privileges
- `.env` is ignored by git and set to `600` permissions
- `LITELLM_MASTER_KEY` is auto-generated if missing
- `./scripts/security-check.sh` is enforced before startup by `./scripts/up.sh`

## Prerequisites

- Docker Desktop with `docker compose`
- macOS with terminal access
- Provider credentials for the models you plan to call
- For local Qwen models: Ollama installed and running

## Setup

1. Create local env file:

```bash
cp .env.example .env
```

2. Fill provider keys/settings in `.env`:

- Required proxy auth key used by all OpenAI clients:
  - `LITELLM_MASTER_KEY` (auto-generated by `./scripts/up.sh` if blank)
- Provider keys as needed:
  - `OPENAI_API_KEY`
  - `ANTHROPIC_API_KEY`
  - `GEMINI_API_KEY`
- Optional strict key enforcement:
  - `LITELLM_REQUIRED_PROVIDER_KEYS` (comma-separated)
  - Current template default: `OPENAI_API_KEY,ANTHROPIC_API_KEY,GEMINI_API_KEY`
  - If you do not use all providers, set only what you actively use
  - Example: `OPENAI_API_KEY,ANTHROPIC_API_KEY`

3. Install secret scanner and git hooks (recommended):

```bash
brew install gitleaks
./scripts/install-hooks.sh
```

4. If using Ollama Qwen models, ensure aliases already exist:

```bash
ollama list | grep -E '^qwen3.5:(9b|27b)-256k[[:space:]]'
```

If missing, create/setup those Ollama model aliases manually

5. Start the proxy:

```bash
./scripts/up.sh
```

6. Check health:

```bash
./scripts/health.sh
```

## Security workflow

- `./scripts/up.sh` auto-runs `./scripts/security-check.sh`.
- Run `./scripts/security-check.sh` manually after any env/config change.
- Run `./scripts/scan-secrets.sh` before commits (git-aware scan).
- Run `./scripts/scan-secrets.sh --full` for strict local scans that include untracked files like `.env`.
- `./scripts/logs.sh` should be used only during active troubleshooting because logs may include prompt/metadata content.

## Phase 1 hardening checklist

- `.env` stays gitignored.
- `.env` stays permissioned at `600`.
- `LITELLM_REQUIRED_PROVIDER_KEYS` matches only providers you actually use.
- `./scripts/security-check.sh` passes before startup.
- `./scripts/scan-secrets.sh` passes before commits.
- Access from outside home network is VPN-only.

## Cross-project usage contract

In any other project shell:

```bash
export OPENAI_BASE_URL=http://127.0.0.1:4000/v1
export OPENAI_API_KEY="$(grep '^LITELLM_MASTER_KEY=' /ABS/PATH/TO/LiteLLM/.env | cut -d '=' -f 2-)"
```

Then use the normal OpenAI client with one of the aliases as `model`.

## Example (Python OpenAI client)

```bash
export OPENAI_BASE_URL=http://127.0.0.1:4000/v1
export OPENAI_API_KEY="$(grep '^LITELLM_MASTER_KEY=' .env | cut -d '=' -f 2-)"
python examples/openai_client_example.py
```

## Phase 1 remote access (current recommendation)

- Keep LiteLLM private on localhost only.
- Use VPN-only remote access (for example Tailscale/WireGuard).
- Do not forward LiteLLM ports from your router.
- Keep `LITELLM_MASTER_KEY` rotated periodically and after major network changes.

## Phase 2 optional path (public exposure, still personal-use only)

If you later want access without VPN, use an HTTPS gateway in front of LiteLLM and keep LiteLLM itself non-public:

- TLS termination + HTTP to HTTPS redirect
- Identity enforcement for your account only (with MFA)
- Rate limits plus request size/time limits
- Deny-by-default policy and alerting on repeated auth failures
- Emergency lockout runbook (disable public route, rotate keys, restart)

## Operations

```bash
./scripts/up.sh
./scripts/down.sh
./scripts/health.sh
./scripts/logs.sh
./scripts/security-check.sh
./scripts/scan-secrets.sh
./scripts/install-hooks.sh
./scripts/rotate-master-key.sh
```

## Key rotation

Rotate proxy auth key:

```bash
./scripts/rotate-master-key.sh
./scripts/down.sh && ./scripts/up.sh
```

All client shells/projects must use the new key value from `.env`.

## Notes

- This setup is intended for personal single-user use.
- Update model IDs in `config/config.yaml` as provider versions change.
- To upgrade LiteLLM securely, update the pinned image digest in `docker-compose.yml`.
